# SMT Files arising from SPARKNaCl for cvc5

This directory contains the SMT files generated by SPARK Pro (Why3 VC Generator)
for verification of the SPARKNaCl cryptographic library, and results showing the
performance of cvc5 on these examples.

The Why3 VCG is very different from other tools. Unlike Boogie and CBMC, Why3
generates roughly "one SMT file per path per assertion". This generates
*lots* of *small* SMT files that can be processed in parallel.

This directory contains 6098 SMT files arising from this library.

## Tool Versions for these experiments

GNATprove (SPARK Pro): 15.1.0-1

Why3: 1.7.1

cvc5: 1.3.3

GNU parallel: 20250122 (get from Homebrew on macOS or apt-get on Linux)

All running on Apple Silicon M1/macOS 26.4.1

## Reproducing the files

1. You'll need GNATprove 15.1.0-1 from [here](https://github.com/alire-project/GNAT-FSF-builds/releases/tag/gnatprove-15.1.0-1)
and put "gnatprove" on your PATH.
2. Checkout the [SPARKNaCl repo](https://github.com/rod-chapman/SPARKNaCl)
3. Put your chosen version of cvc5 on PATH
4. Genereate SMT files with

```
cd SPARKNaCl
gnatprove -Psparknacl --prover=cvc5 --timeout=5 --steps=0 --report=fail --debug-save-vcs
```

5. Generated SMT files end up in `obj/gnatprove` directory

## Results

See the doall.sh script in this directory. This runs cvc5 with a 5-second
timeout on all files. Latest results are in `res.txt`

### cvc5 1.3.3

With default options, cvc5 1.3.3 finds "unsat" on 6039 of these, with 59 timing out.
Results are in `res1.3.3.txt`

# Conclusions (so far...)

Good result. We are *close* to a full proof of an entire cryptographic algorithm/library
using only a single prover.

Performance is also good. Using 10 processor cores on a Apple Silicon M1 MacBook Pro,
all 6098 proofs can be re-generated in 139 seconds real time. Reproduce with

```
time ls *.smt2 | parallel -j10 --bar cvc5 --tlimit=5000 {}
```

Using 96 cores on an EC2 r8g.24xlarge-metal instance completes the run in 32 seconds.

# Experiments with Z3 and Portfolio

## Running Z3 only

Running Z3 alone with a 5s timeout over these files produces 5898 "unsat" and 200 timeouts.

BUT... these files are generated by Why3 specifically for cvc5, so do not favour Z3.

## Portfolio using Z3 and cvc5

The `doz3cvc5.sh` script runs both Z3 and cvc5 over these SMT files and resords results
for both in `resz3cvc5.txt`

This shows BOTH z3 and cvc4 return "unsat" for 5854 of 6098 proofs, leaving 244.

The remaining 244 break down as follows:

```
z3 timeout cvc5 unsat:  184
z3 timeout cvc5 timeout: 16
z3 unsat   cvc5 timeout: 44
Total:                  244
```

That leaves 16 "hard ones" - listed in the file `hardones.txt` in this directory.

On those, 15 comes from just one function in the Ed25519 signing algorithm `SPARKNaCl.Sign.ModL` - one
of the most complex bits of code in the library. See [sources here](https://github.com/rod-chapman/SPARKNaCl/blob/8e3cc2e6a67826cbfb1e36559238e03cb024a706/src/sparknacl-sign.adb#L240)

# CVC5 5-way portfolio

Analysis of the "hard 16" with Claude Opus suggests that the following portfolio of 5 sets
of command-line options is effective:

```
cvc5 --tlimit=5000
cvc5 --tlimit=40000 --finite-model-find --enum-inst
cvc5 --tlimit=20000 --enum-inst-interleave
cvc5 --tlimit=20000 --decision=internal
cvc5 --tlimit=20000 --miniscope-quant=off
```

See the script `cvc5p5.sh` which runs this portfolio (in series) on a given SMT file:

```
#!/bin/zsh
R1=`cvc5 --tlimit=5000 $1`
R2=`cvc5 --tlimit=40000 --finite-model-find --enum-inst $1`
R3=`cvc5 --tlimit=20000 --enum-inst-interleave $1`
R4=`cvc5 --tlimit=20000 --decision=internal $1`
R5=`cvc5 --tlimit=20000 --miniscope-quant=off`
echo $1,$R1,$R2,$R3,$R4,$R5
B=`basename $1 .smt2`
echo $1,$R1,$R2,$R3,$R4,$R5 >$B.p5res
```
Later, we will refer to those options as "DEF" (for "default"), "FMFEI", "EII", "DI", and "MQO" respectively.

We can then run that script over all the SMT files with the script `p5allpar.sh`

```
#!/bin/zsh
ls *.smt2 | parallel -j8 --bar ./cvc5p5.sh {}
cat *.p5res | sort >all.p5res
```

to get `all.p5res` containing the results of all 5 sets of options on all files.

Further analysis with Excel shows the number of "unsat"s achieved by each member of the portfolio:

```
DEF:   6039 (5 second timeout)
FMFEI: 5840 (40 second timeout)
EII:   6035 (20 second timeout)
DI     6076 (20 second timeout)
MQO    6078 (20 second timeout)
```

Suggesting a "first-past-the-post" sequential portfolio in the following order:

DEF then MQO then DI then EII then FMFEI

might offer the best performance.

This is embodied in the `cvc5p5seq.sh` script, which terminates when
first member of the portfolio returns "unsat", rather than running all
of them.

Results are in `allseq.p5res`. That says the number of "unsats" returned by each
member of the portfolio as:

```
DEF:   6037
MQO:     41
DI:       6
EII:      1
FMFEI:   13

TOTAL: 6098
```

On an Apple M1 Macbook Pro with 10 CPU cores running, this portfolio completes the proof in 6 minutes real time.

On an EC2 r8g.24xlarge-metal instance using 96 cores, the proof completes in 2 minutes.
